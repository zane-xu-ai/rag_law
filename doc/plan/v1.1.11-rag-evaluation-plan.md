## 2. 金标评测集（三类指标的共同前提）

### 2.1 数据格式（建议）

为每条样本至少包含：

| 字段 | 用途 |
| --- | --- |
| `query_id` | 稳定主键 |
| `query` | 用户问题文本 |
| `gold_chunk_ids` | 与 ES `_id` 或业务 `chunk_id` 一致的可判重列表（**Recall@k** 判「是否命中任一 gold」） |
| `gold_primary_chunk_id`（可选） | 标注者认定的**首个**最应展示块；**无则默认取 `gold_chunk_ids[0]`** 供 **MRR** 使用 |

条数建议：**不少于 50 条**（P0）；扩展至 100～200 条用于回归更稳。与 [v1.1.8](v1.1.8-scheme-d-eval.md) 第 4 节一致，题型需覆盖法条直问与跨条综合问。

### 2.2 与当前实现对齐

- 检索：生产路径为 **query 向量 → ES kNN → top-k**，k 与 **`RETRIEVAL_K`**（及 WebUI 所选 k）对齐时，**Recall@k / MRR 中的 k 须与之一致**，否则须在报告中单列实际 k。
- 生成：使用与线上一致的 **system/user 拼装**（见 `qa.messages.build_messages`）及同一 **LLM**；RAGAS 的 `contexts` 取 **top-k 片段正文列表**（与喂给模型的顺序一致）。

## 3. Recall@k（必测）

### 3.1 定义（本仓库采用）

对单条 query：若 **top-k 检索结果** 的 chunk 标识集合与 **`gold_chunk_ids` 交集非空**，则该条 **命中**，记 1；否则记 0。

**Recall@k** = 全体评测 query 上该二值标签的 **算术平均**（等价于「至少命中一个 gold 的 query 比例」）。

若存在多条 gold 且需统计「gold 被覆盖比例」，须在报告中**另起子指标**命名（如 `gold_coverage@k`），勿与上述标准 Recall@k 混用。

### 3.2 实施要点

- **固定索引**：对比切块/重排策略时，为每个策略建独立 `ES_INDEX` 或使用 `scope`/版本字段；见 [v1.1.8](v1.1.8-scheme-d-eval.md) 第 4 节。
- **输出**：`recall_at_k.json` 或表格：`k`、`n_queries`、**Recall@k**、按 `source_file` 分层统计（可选）。

---

## 4. MRR（必测）

### 4.1 定义（本仓库采用）

对单条 query：在 **全量排序列表**（或实践上 **截断至前 `k_max`**，如 `k_max = max(20, RETRIEVAL_K×2)`，须在报告中声明）中，找到 **第一个**出现在列表中的 **主相关块**：

- 主相关块 id = `gold_primary_chunk_id`；若缺省，则取 **`gold_chunk_ids[0]`**。

该条 query 得分 = **1 / rank**（`rank` 从 1 起）；若截断范围内未出现，得分为 **0**。

**MRR** = 全体 query 得分的 **算术平均**。

### 4.2 实施要点

- 与 Recall@k 使用 **同一检索实现、同一快照**。
- 输出：`mrr.json` 或表格：**MRR**、截断深度、`rank` 分布直方图（可选）。

## 5. RAGAS 框架（必测）

### 5.1 须覆盖的指标（最低集）

在每条样本上，除 `query`、**检索得到的 `contexts`**（字符串列表，与模型调用一致）外，还须得到 **`answer`**（与评测用 LLM 一次生成的正文）。对以下 RAGAS 指标**全部计算并在报告中给出均值**（具体 API 以所选 `ragas` 版本为准）：

| RAGAS 指标 | 含义（简述） | 备注 |
| --- | --- | --- |
| **Context precision** | 检索上下文中有用比例（与答案相关性） | 依赖评判模型 |
| **Context recall** | 上下文相对「标准答案」的覆盖程度 | 金标若仅有 chunk id，可用 gold 片段拼接或参考答作 `reference` |
| **Faithfulness** | 答案是否可由上下文推出 | 法律场景必看；与 [v1.0.6](v1.0.6-mvp-acceptance-plan.md) 人工 C3 互补 |
| **Answer relevancy** | 答案与问题的相关性 | |

可选扩展：**Answer correctness**（相对参考答）、**Answer similarity** 等；若启用须在报告中说明参考答来源。

### 5.2 实施要点

- **依赖**：评测环境单独 `uv` 组（如 `optional-dependencies` 增加 `eval` / `ragas`），勿与生产 serving 混用同一约束时可简化。
- **评判模型**：与业务 LLM 可分离；须在报告中写明 **评判用模型名、温度、版本**，保证可复现。
- **与 Recall@k / MRR 同批**：同一 `query`、同一检索快照、同一 `contexts` 再生成 `answer` 后跑 RAGAS，避免「检索用 A 索引、RAGAS 用 B 上下文」。
- **人工抽检**：RAGAS 分数升高不豁免 [v1.0.6](v1.0.6-mvp-acceptance-plan.md) 抽检；对 **Faithfulness** 最低的若干条强制人工打开原文核对。

### 5.3 参考答（Context recall 等）

- 若有 **人工撰写的短参考答**：直接作为 `reference`。
- 若仅有 **gold chunks**：可将 gold 片段按顺序拼接为 `reference`（长度截断策略须固定并写入报告）。

## 6. 推荐执行顺序（一次完整评测轮次）

1. 构建或更新 **金标集**（第 2 节）；冻结 `ES_INDEX` 与索引内文档版本。  
2. 跑 **Recall@k**（第 3 节）与 **MRR**（第 4 节）——可同一脚本一次检索、双指标输出。  
3. 对同批 query 用固定 prompt 调 LLM 得到 **answer**，组装 **RAGAS** 数据集并跑第 5 节指标。  
4. 汇总 Markdown/HTML 报告：**Recall@k**、**MRR**、各 **RAGAS** 均值 + 环境指纹 + 失败样例列表。  
5. 抽样执行 [v1.0.6](v1.0.6-mvp-acceptance-plan.md) 清单，记录与 RAGAS 不一致的案例。

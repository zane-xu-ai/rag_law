# v1.1.3 Token 成本观测：用量归因、价格映射与日成本核算

| 属性 | 说明 |
| --- | --- |
| 文档版本 | v1.1.3 |
| 状态 | 计划（待实施） |
| 前置 | v1.1.0 日志统一、v1.1.1 QA 监控已落地（`logs/monitor.log` / ES 监控索引） |
| 适用范围 | 统计 LLM token 消耗（模型名、输入/输出/思考 token）、写入日志与监控，并按价格计算单次与日成本 |

---

## 1. 目标与范围

在现有 QA 监控（耗时、检索、成功率）基础上，补齐**成本维度**：

- 每次请求可追踪：`model`、`input_tokens`、`output_tokens`、`reasoning_tokens(如有)`、`cached_tokens(如有)`、`cost_usd`。
- 可按日聚合：请求数、token 总量、成本总额、按模型成本分布。
- 与已有 `monitor.log` / ES / Grafana 对齐，尽量复用现有观测链路与字段风格。

**不在本版必选项**：

- 严格财务账单对账（云厂商账单通常有延迟与舍入差异）。
- 多租户成本分摊（团队/用户维度）与预算审批流。

---

## 2. 企业级方案对比（建议选型）

## 2.1 方案 A：应用内埋点 + 自建成本表（推荐起步）

在应用代码中直接读取供应商响应里的 usage 字段，写入 `monitor` 文档；价格通过配置/表维护。

**优点**：

- 与业务上下文最完整（可携带 `conversation_id`、`query_fp`、检索参数）。
- 实施快，改动集中在本仓库（`qa.streaming` + `conf.monitor` + `conf.settings`）。
- 可与现有 v1.1.1 监控无缝融合。

**缺点**：

- 要维护价格映射与模型版本变化。
- 多供应商差异需要适配层。

## 2.2 方案 B：LLM 网关 / Proxy 统一计量（企业常见）

使用 LiteLLM Proxy、Helicone、OpenRouter Enterprise、API Gateway + 自定义插件等，在网关层统一记录 token 与成本。

**优点**：

- 多模型多供应商统一口径，便于治理与审计。
- 可统一限流、配额、预算告警。

**缺点**：

- 架构复杂度上升；改造网络拓扑与凭据管理。
- 若网关不带业务上下文，仍需应用侧补充关联字段。

## 2.3 方案 C：可观测平台集成（Datadog/New Relic/Langfuse/LangSmith）

把 token/cost 作为 span 属性或事件上报，直接使用平台看板与告警。

**优点**：

- 企业成熟度高，告警/报表能力强。
- 更容易跨服务关联（trace 级别）。

**缺点**：

- 商业成本与数据出境合规需评估。
- 仍要做字段规范与采样策略。

## 2.4 本仓库建议路径

1. **短期（v1.1.3）**：采用方案 A，先跑通“可核算、可看板、可告警”。
2. **中期（v1.1.4+）**：若模型供应商增多，再引入网关方案 B。
3. **长期**：需要统一企业可观测治理时接入方案 C。

## 3. 数据契约（新增字段建议）

在当前 `build_qa_monitor_document` 基础上新增 `llm_usage` 与 `cost` 分组字段（命名可微调，但需稳定）：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `llm.model` | string | 实际调用模型名（返回值优先，其次配置 `MODEL_NAME`） |
| `llm.provider` | string | 供应商标识（openai/dashscope/azure_openai/...） |
| `llm.usage.input_tokens` | int | 提示词 token（query + context + system/user） |
| `llm.usage.output_tokens` | int | 输出 token |
| `llm.usage.total_tokens` | int | 总 token |
| `llm.usage.reasoning_tokens` | int\|null | 深度思考 token（若供应商提供） |
| `llm.usage.cached_tokens` | int\|null | 缓存命中 token（若供应商提供） |
| `llm.cost.currency` | string | 货币（建议默认 `USD`） |
| `llm.cost.input_cost` | number | 输入 token 成本 |
| `llm.cost.output_cost` | number | 输出 token 成本 |
| `llm.cost.reasoning_cost` | number\|null | 思考 token 成本（若计费项独立） |
| `llm.cost.total_cost` | number | 本次请求总成本 |
| `llm.cost.price_version` | string | 价格表版本（用于回溯） |

**兼容性说明**：

- 各供应商 usage 结构不一致：字段提取采用“能力探测 + 容错默认 0/None”。
- 只记录长度/指纹，不记录 query 原文，沿用 v1.1.1 隐私约束。

## 4. 价格映射与核算规则

## 4.1 价格配置来源（建议）

新增价格配置（建议 YAML/JSON + 环境版本号）：

- 例：`conf/llm_price_table.json` 或 `doc/cost/llm_prices.example.json`
- key 维度：`provider + model + effective_from`
- value：`input_per_1m`、`output_per_1m`、`reasoning_per_1m(可选)`、`cached_input_per_1m(可选)`

## 4.2 计算公式

- `input_cost = input_tokens / 1_000_000 * input_per_1m`
- `output_cost = output_tokens / 1_000_000 * output_per_1m`
- `reasoning_cost` / `cached_cost` 同理（若价格项存在）
- `total_cost = input_cost + output_cost + reasoning_cost + ...`

## 4.3 精度与舍入

- 内部存储建议 `Decimal`（或至少保留 8~10 位小数）；
- 展示层按 4~6 位小数；
- 聚合层（日成本）最后再舍入，避免累计误差。

## 5. 日志与监控落地（对接现有仓库）

## 5.1 代码接入点

- [`src/qa/streaming.py`](../../src/qa/streaming.py)：在 `chat.completions.create(..., stream=True)` 流结束后提取 usage，并在成功/异常分支统一写监控记录。
- [`src/conf/monitor.py`](../../src/conf/monitor.py)：扩展 `build_qa_monitor_document` 支持 `llm` 与 `cost` 字段。
- [`src/conf/settings.py`](../../src/conf/settings.py)：增加成本配置项（如 `LLM_PRICE_TABLE_PATH`、`LLM_PRICE_VERSION`、默认货币等）。

## 5.2 索引与看板

- 继续写入现有监控索引（当前已使用 `rag-law-monitor-v2`），并增加新字段 mapping。
- Grafana 新增面板：
  - 每日 `sum(llm.cost.total_cost)`
  - 按模型分组成本
  - input/output/reasoning token 趋势
  - 单次成本 P50/P95（识别异常高成本请求）

## 5.3 告警建议

- 日成本超过阈值（如 > X USD）告警。
- 某模型单位请求成本异常上升告警（可能是 prompt 变长或模型切换）。

## 6. 实施阶段建议

## Phase 1（MVP，1~2 天）

- 单供应商（当前主供应商）usage 提取；
- 记录 `model/input/output/total_tokens/total_cost`；
- Grafana 出“日成本 + 模型分布”两个面板。

## Phase 2（增强，2~4 天）

- 增加 `reasoning_tokens/cached_tokens` 兼容提取；
- 价格表版本化与热更新（重启生效即可）；
- 完善异常分支（无 usage 返回时的兜底记录）。

## Phase 3（企业化，可选）

- 引入网关统一计量（LiteLLM/Helicone 等）；
- 增加多租户成本标签（team/project/user_id）；
- 成本预算与配额策略闭环。

---

## 7. 验收标准

- 单次 QA 请求日志中可见：模型名、input/output token、总成本；
- ES 可按日聚合成本，Grafana 可展示每日成本趋势；
- 若供应商返回 reasoning token，可单独展示并计费；
- 不记录敏感原文（仅长度/指纹）。

---

## 8. 风险与规避

- **供应商字段变更风险**：usage 解析器做版本兼容与容错。
- **价格变更风险**：价格表带 `price_version` 与生效时间，避免历史回放混乱。
- **对账误差**：与账单存在小时级延迟/舍入差异，文档中明确“监控估算值”语义。

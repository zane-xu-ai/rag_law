## 1. 目标

1. 在实现正式 `es_store` 模块前，用**可重复脚本或集成测试**验证本地 Elasticsearch **可连通、可建索引、可写入、可检索**（先可用 `match_all` / 简单 term；可选再验证 `knn`）。
2. 统一**索引命名**（`rag_law_doc_*`、`rag_law_log_*` 及跨项目前缀），避免与「其它 RAG / Agent」索引混用。
3. 实现 MVP §6 所述 **mapping、bulk、kNN 检索**，与 [`src/embeddings`](../../src/embeddings/) 的 **`dense_dimension`**、**L2 + cosine** 一致。
4. **（规划）** 扩展 chunk 文档字段：支持**可追溯元数据**（业务文档 id、内容哈希、页码等）与 **`chunk_type`（默认文本）**，默认值在**应用层**写入 `_source`，与 `TextChunk` 对齐。

**不在本版本（可后续迭代）**：混合检索（BM25 + 向量 + RRF）、MySQL 元数据、生产级 ILM/多集群。

## 2. 索引命名约定

| 前缀 | 用途 | 示例 |
| --- | --- | --- |
| `rag_law_doc_` | 本仓库 **文档块 / 正文检索** 类索引（chunk、`text`、`embedding`、偏移等） | `rag_law_doc_chunks` 或 `rag_law_doc_chunks_v1` |
| `rag_law_log_` | 本仓库 **日志、审计、入库运行记录**（预留，MVP 可不建） | `rag_law_log_ingest` |
| `rag_` | **其它** RAG 项目或实验（非本仓库专属命名空间时） | `rag_demo_xxx` |
| `agent_` | **Agent** 相关（轨迹、工具调用等，与 chunk 索引分离） | `agent_session_xxx` |

**配置**：继续使用 [`src/conf/settings.py`](../../src/conf/settings.py) 中 `ES_HOST`、`ES_PORT`、`ES_INDEX`、`ES_USER` / `ES_PASSWORD` / `ES_USE_SSL`。建议将默认 **`ES_INDEX`** 从 `rag_law_chunks` **迁移为** `rag_law_doc_chunks`（或带版本后缀 `rag_law_doc_chunks_v1`），与上表一致；旧名可在迁移说明中标注为废弃别名。

若日后单独存日志索引，可新增 `ES_LOG_INDEX` / `RAG_LAW_LOG_INDEX`（实施时再定）。

## 3. 当前 chunk 索引字段（已实现）

实现位置：[`src/es_store/mapping.py`](../../src/es_store/mapping.py)、[`src/es_store/store.py`](../../src/es_store/store.py)。

| 字段 | ES 类型 | 说明 |
| --- | --- | --- |
| `text` | `text` | chunk 正文 |
| `embedding` | `dense_vector` | `dims` = 模型 [`dense_dimension`](../../src/embeddings/bge_m3.py)；`index: true`；`similarity: cosine`（与向量层 L2 归一化一致） |
| `source_file` | `keyword` | 来源「文件名」类标识 |
| `source_path` | `keyword` | 来源路径（如仓库内路径、notebook 引用等） |
| `chunk_index` | `integer` | 同一来源文件内块序号（从 0 递增） |
| `char_start` / `char_end` | `integer` | 相对**该次切分所用全文**的字符偏移（Python 切片语义） |
| `chunk_type` | `keyword` | 模态，默认 `text`（`apply_chunk_source_defaults`） |
| `mime_type` / `doc_type` / `domain` | `keyword` | 与 `TextChunk` 一致，见 [`chunk_defaults.py`](../../src/es_store/chunk_defaults.py) |
| `source_doc_id` | `keyword` | 业务文档 id；默认 `""` |
| `source_sha256` | `keyword` | 源或正文哈希；默认 `""` |
| `page_start` / `page_end` | `integer` | 可选；PDF 等 |
| `extra` | `flattened` | 长尾键值；可选 |

**文档 `_id`（非 `_source`）**：`chunk_document_id(source_file, chunk_index)` → `"{source_file}:{chunk_index}"`，用于 bulk 幂等；**不是**业务侧稳定「文档 UUID」。若多文档共用一个 `source_file` 展示名，未来可评估是否把 `doc_id` 纳入 `_id` 策略（见 §4.4）。

**与 [`TextChunk`](../../src/chunking/split.py) 的对齐**：mapping 已含 `chunk_type`、`mime_type`、`doc_type`、`domain`、`source_doc_id`、`source_sha256`、页码、`extra` 等（见 §4）；`bulk_index_chunks` 经 `apply_chunk_source_defaults` 补齐默认值，并校验 `text`、`embedding` 维度及建 `_id` 所需字段。笔记本 [`notebooks/c04_es.ipynb`](../../notebooks/c04_es.ipynb) 演示从 `TextChunk` 写入上述字段。

**查询**：`knn` 在 `embedding` 上，`k` 取自 `RETRIEVAL_K`；扩展后可 `filter`（如 `source_doc_id`、`chunk_type`）。

**客户端**：Python `elasticsearch` 包与 **Server 8.x/9.x** 兼容（以官方矩阵为准）。

## 4. chunk 元数据扩展（设计与实现）

本节说明可追溯字段、默认值与长尾策略；**mapping 与 `apply_chunk_source_defaults` 已落地**（见 [`mapping.py`](../../src/es_store/mapping.py)、[`chunk_defaults.py`](../../src/es_store/chunk_defaults.py)）。入库管线仍需在能计算时写入 `source_sha256`、页码等。

### 4.1 需求与字段覆盖

| 诉求 | 字段 / 状态 | 说明 |
| --- | --- | --- |
| 文档「展示名」 | `source_file`（及可选 `title` 待加） | 展示名通常够用 |
| 稳定 **文档 id** | `source_doc_id`（`keyword`）；默认 `""` | 与 `TextChunk.source_id` 对齐写入；业务 UUID 由上游赋值 |
| **内容 / 源文件 hash** | `source_sha256`（`keyword`）；默认 `""` | 在读源或整篇正文后计算并写入 |
| **页码** | `page_start` / `page_end`（`integer`，可选） | Markdown 无页可省略；PDF 管线填充 |

### 4.2 `chunk_type`（chunk 数据类型，默认文本）

- **含义**：chunk 的模态/类型（文本、表格、标题、OCR 等），便于检索策略与展示分支。
- **类型**：`keyword`。
- **应用层默认值**：`"text"`。Elasticsearch **不在 mapping 中声明 default**，由入库代码写入。
- **其它取值示例**（不必做成严格枚举）：`table`、`heading`、`ocr`、`code`；未识别时仍写 `text`，避免空值。

### 4.3 与 `TextChunk` 对齐的默认值（写入 `_source`）

| 字段（建议进 ES） | 与 `TextChunk` | 建议默认值 |
| --- | --- | --- |
| `chunk_type` | 新增 | `text` |
| `mime_type` | 已有 | `text/markdown` |
| `doc_type` | 已有 | `law_md` |
| `domain` | 已有 | `law` |
| `source_path` | 已有 | 无则 `""` 或与 `source_file` 一致 |
| `source_doc_id` | 业务主键 | 无则 `""`（需 filter 时优先空串而非缺键） |
| `source_sha256` 等 | 可选 | 无则 `""` 或省略 |

可选后续：`language`（`keyword`，如 `zh` / `und`），多语料再上。

### 4.4 扩展字段策略：核心进 mapping，长尾进 `extra`

1. **优先写进显式 mapping 的**：高频、需 **filter / 聚合** 的字段——`chunk_type`、`source_doc_id`、`source_sha256`、页码、`mime_type`、`doc_type`、`domain`。
2. **长尾、各源不一致的**：用 **`extra`（`object` 或 `flattened`）** 或 `labels`（`keyword` 数组），**有稳定需求后再**提升为顶层字段，避免 mapping 膨胀。
3. **适合「遇到再加」的**：法院层级、案号细分、法规编号等——待某条入库链路能稳定产出再定 schema。

### 4.5 实施要点（落地时）

- **扩展** [`chunk_index_mappings`](../../src/es_store/mapping.py)：新增上述 `keyword` / `integer` 等；`extra` 按需选型 `flattened`（键多且动态）或 `object`。
- **入库**：[`bulk_index_chunks`](../../src/es_store/store.py)、[`notebooks/c04_es.ipynb`](../../notebooks/c04_es.ipynb) 等组装 `dict` 时填充默认值，与 `TextChunk` 一致。
- **切分**：[`iter_chunks_for_text`](../../src/chunking/split.py) / `iter_file_chunks` 在能提供上游时写入 `extra` 或未来新字段（hash、页码等）。
- **索引迁移**：显式 `properties` 下新增字段通常需 **新建索引或 `recreate` + 全量重导**；旧索引需 **reindex** 时单独说明。

## 5. 阶段 A：本地 ES 读写冒烟

**目的**：在写 `src/es_store` 之前，先排除网络、权限、版本不匹配问题。

**推荐步骤**：

1. 本机启动 ES（MVP 建议镜像：`docker.elastic.co/elasticsearch/elasticsearch:9.3.2`，单节点；可在 [`doc/storage/`](../../doc/storage/) 提供示例 `docker-compose.elasticsearch.yml`，**不强制**作为生产配置）。
2. 在 [`pyproject.toml`](../../pyproject.toml) 增加 **`elasticsearch`** 依赖（并可选 **extra**，例如 `es` 或与 `ingest` 组合），避免未装 ES 客户端时破坏仅 `conf`/`chunking` 的安装路径。
3. 提供 **`scripts/es_smoke_test.py`**（或等价模块）：  
   - `ping`；  
   - 创建**临时索引**（如 `rag_law_doc_smoke` 或带时间戳）；  
   - 写入 1～2 条文档 → `refresh` → `search` 校验命中；  
   - 可选删除索引。  
   第一条路径**不依赖**真实向量。  
   若环境已就绪，可额外写入一条 **dummy `dense_vector`**（维度与 `dense_dimension` 一致）并执行 **`knn`** 查询。
4. 可选：**pytest** 集成（`@pytest.mark.integration`），无 ES 时跳过；与 CI 策略一致（CI 可不装 ES）。

**验收**（阶段 A）：脚本/测试无报错完成「建 → 写 → 读 →（可选）删」；若启用 kNN 路径，mapping 中 `dense_vector` 与查询参数与官方文档一致。

## 6. 阶段 B：存储层实现

| 建议路径 | 职责 |
| --- | --- |
| `src/es_store/`（或 `src/elasticsearch/`） | 获取客户端（复用 `settings.es_url` 等）；`ensure_index(mapping)`；`bulk_upsert` chunks；`search(query_vector, k)`（内部组装 `knn` DSL） |

**函数契约（建议）**：

- `search(query_vector: list[float], k: int)` → 返回命中列表（含 `_id`、`score`、`_source` 或投影字段）。
- `k` 默认来自 `settings.retrieval_k`，允许调用方覆盖。

**依赖**：[embeddings](v1.0.2-bge-m3-embedding-plan.md) 已产出归一化向量；`dims` 与 `dense_dimension` 一致。

## 9. 风险与注意

- **TLS/认证**：若本地 ES 开启安全，使用 `ES_USER` / `ES_PASSWORD` / `ES_USE_SSL`（与现有 Settings 一致）。
- **客户端与 Server 主版本**：需与集群主版本匹配（见 Elastic 官方矩阵）。
-  **`transformers` / `FlagEmbedding`**：与向量层无关，但同一环境共存时注意依赖不冲突（见 [embedding 计划](v1.0.2-bge-m3-embedding-plan.md) 与当前 `pyproject` 锁定）。
- **mapping 变更**：§4 字段上线后，已有索引需按 §4.5 处理，避免与线上数据假设不一致。

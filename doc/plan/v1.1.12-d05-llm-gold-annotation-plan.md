## 1. 范围与前提

### 1.1 输入数据

- **目录**：`data/chunk_md/d05_heading_presplit_document_segmentation/`（相对项目根；若本地尚未导出，先按 [v1.1.10](v1.1.10-md-heading-presolve-scheme-d.md) / d05 脚本生成该目录）。
- **形态**：每个源法规对应**一个** `.md` 文件；文件内多块正文由**固定分隔条**拼接（与导出脚本一致），**不是**「一行一块」的 JSONL。

### 1.2 切块还原（与线上一致）

对目录下每个 `*.md` 文件：

1. 取文件名 **`source_file`** = 该文件 `name`（与入库时 `TextChunk.source_file` 一致，通常为 `md3__宪法.md` 等形式）。
2. 按 `chunking.md_heading_presplit.default_heading_presplit_separator()` 的**完整字符串**切分全文，得到 `pieces[]`。
3. 第 `i` 段（从 0 起）即 **`chunk_index = i`** 的块正文（可做 strip 与否须在标注流水线与后续入库评测中**统一**，建议：**切分用原始分隔逻辑，单块正文 trim 首尾空白后再送模型**）。
4. **与 ES `_id` 对齐**（必守）：`gold_chunk_id = f"{source_file}:{chunk_index}"`，即 `es_store.store.chunk_document_id(source_file, chunk_index)` 的返回值，保证与当前索引幂等写入规则一致。

### 1.3 输出在 v1.1.11 中的用途

| v1.1.11 字段 | 本计划产出 |
| --- | --- |
| `query_id` | 由 `source_file`、`chunk_index`、同块下 query 序号生成稳定 id（建议：`sha256` 短前缀或 `"{stem}__c{idx}__q{n}"`） |
| `query` | 模型生成的用户问题（中文、法律检索口吻） |
| `gold_chunk_ids` | 至少含当前块 id；若模型生成「需多块」题可含相邻块 id（须在 schema 中显式列出） |
| `gold_primary_chunk_id` | 可选；多 gold 时由模型或规则指定 MRR 主相关块，缺省则 v1.1.11 取 `gold_chunk_ids[0]` |
| `reference`（RAGAS 可选） | 模型生成的短参考答，或 gold 片段摘要；须在评测脚本与报告中固定策略 |

## 2. 标注任务定义（按 chunk）

对**每一个** `(source_file, chunk_index, chunk_text)`：

1. **主任务**：生成 **1～2 条** 自然语言 **query**，使得在理想检索下**应优先召回本块**（及必要时相邻块）；问题类型覆盖：
   - **直述型**：针对本块核心法条或概念的直接提问；
   - **场景型**：贴近用户表述的综合问，但答案依据仍主要落在本块（可标注 `gold_chunk_ids` 含最多 **3** 个 id，避免标注噪声过大）。
2. **负向约束**（写入 prompt）：不得虚构不存在的条文号；若块内无明确条号，问题中不得强行编造具体「第×条」。
3. **可选输出**（推荐用于 RAGAS）：
   - `reference`：1～3 句中文短答，**仅依据本块（及已列 gold 块）**可答内容撰写，便于 Context recall 等指标。

每条模型输出经校验后，展开为 v1.1.11 的**一行样本**（一 query 一行；同一 chunk 的第二条 query 另一行）。

## 3. 大模型调用与工程约定

### 3.1 模型与参数

- 使用与 `.env` 中 **OpenAI 兼容**接口（如 DashScope）均可；**标注用模型**可与**评测用生成模型**分离，但须在**标注元数据**中记录：`annotator_model`、`base_url`（脱敏）、`temperature`（建议 ≤0.3）、`prompt_version`。
- **批量**：按文件或按固定 batch（如每请求 8～16 个 chunk 摘要+生成）切分，控制 token；**单 chunk 超长**时先截断送标注（例如 head+tail 各 2000 字并标明截断），完整正文仍写入落盘的 `chunk_text_sha256` 或路径引用以便审计。

### 3.2 输出格式（机器可解析）

- 优先 **JSON Lines**：每行一个对象，字段至少：

```json
{
  "query_id": "md3__宪法__c12__q0",
  "query": "……",
  "gold_chunk_ids": ["md3__宪法.md:12"],
  "gold_primary_chunk_id": "md3__宪法.md:12",
  "reference": "……",
  "source_file": "md3__宪法.md",
  "chunk_index": 12,
  "annotator_model": "…",
  "prompt_version": "v1.1.12-0"
}
```

- **`gold_chunk_ids` 内字符串**必须与第 1.2 节 `chunk_document_id` 规则完全一致（含 `.md` 等与磁盘文件名一致的后缀）。

### 3.3 校验规则（脚本强制执行）

- JSON 可解析；`query` 非空；`gold_chunk_ids` 非空且每个 id 可拆成 `(source_file, int)` 且该 `source_file` 存在于 d05 目录、**chunk_index 在切分长度范围内**。
- `gold_primary_chunk_id ∈ gold_chunk_ids`（若提供）。
- **去重**：`query` 与已有集合归一化（去空白、小写可选）后相似度过高则丢弃或合并（阈值可配置）。

### 3.4 建议落盘路径

- 原始模型输出：`data/eval/gold/d05_llm_raw/`（按日期或 commit 分子目录）。
- 校验合并后：`data/eval/gold/d05_llm_queries.jsonl`（或 `.json` 数组），并在同目录写 `manifest.json`（d05 目录哈希、文件列表、参数、条数）。

## 4. 质控与人审

| 环节 | 比例 / 触发 | 说明 |
| --- | --- | --- |
| 自动校验 | 100% | 第 3.3 节 |
| 人工抽检 | ≥5% 或每文件至少 2 条 | 核对 query 与 chunk 是否匹配、是否违规编造条号 |
| 争议样本 | 全量记录 | 人审修改后写入 `*_human_fix.jsonl` 并保留原行 id |

## 5. 执行顺序（建议）

1. 确认 d05 导出目录齐全且与即将入库的 **`chunk_version` / 索引 scope** 一致（见 [v1.1.8](v1.1.8-scheme-d-eval.md) 第 4 节「固定索引」思路）。  
2. 编写或使用脚本：**枚举 chunk**（第 1.2 节）→ **构造 prompt** → **调用 LLM** → **校验** → **写出 jsonl**。  
3. 跑人工抽检，修正明显错标。  
4. 将 `data/eval/gold/d05_llm_queries.jsonl`（或经批准路径）登记为 v1.1.11 评测的**正式金标版本**，评测报告引用 `manifest.json`。

---

## 6. 与「全量 chunk」的关系说明

- **全量**：指 d05 目录下**每个导出文件中的每一个 chunk 段**均至少生成 **1 条**主 query（可再加第 2 条变体）；不是「每个字节」标注。
- 若个别块为纯表格/噪声，允许在 manifest 中标记 **`skipped_reason`**，并在人审中确认；**跳过比例**建议控制在 **<2%** 且需说明。

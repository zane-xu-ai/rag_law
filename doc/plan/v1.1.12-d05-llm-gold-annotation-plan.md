## 1. 范围与前提

### 1.1 输入数据

- **目录**：`data/chunk_md/d05_heading_presplit_document_segmentation/`（相对项目根；若尚未导出，先按 [v1.1.10](v1.1.10-md-heading-presolve-scheme-d.md) / d05 脚本生成）。
- **形态**：每个源法规对应**一个** `.md` 文件；块与块之间由**固定分隔条**拼接。

### 1.2 切块还原（与线上一致）

对目录下每个 `*.md` 文件：

1. **`source_file`** = 该文件 `name`（与入库 `TextChunk.source_file` 一致）。
2. 按 `chunking.md_heading_presplit.default_heading_presplit_separator()` 的**完整字符串**切分全文，得到 `pieces[]`。
3. 第 `i` 段（从 0 起）为 **`chunk_index = i`**；单块正文建议 **trim** 后再送模型（全流水线须统一）。
4. **ES `_id` 对齐**：`gold_chunk_id = f"{source_file}:{chunk_index}"`（`chunk_document_id`）。

### 1.3 输出在 v1.1.11 中的用途

| v1.1.11 字段 | 本计划产出 |
| --- | --- |
| `query_id` | 稳定主键（见第 4.2 节） |
| `query` | 模型生成的用户问题 |
| `gold_chunk_ids` / `gold_primary_chunk_id` | 与评测集 schema 一致 |
| `reference` | 可选，供 RAGAS 等使用 |

## 2. 标注任务定义（按 chunk，模型自判题量）

对**每一个** `(source_file, chunk_index, chunk_text)` 调用一次结构化生成（见第 3 节）：

1. **模型须自判**：当前块是否足以支撑**有意义、可检索对齐**的用户问题。
   - **仅标题、极短过渡句、无实质法条/事实**的块：允许输出 **0 条** query，并给出 **`skip_reason`**（枚举或短语文，如 `title_only`、`too_short`、`navigational`）。
   - **有实质内容**的块：输出 **1～5 条** query（**不必凑满 5**；条数按信息量与多样性需要而定）。
2. **题型**：直述型（指向本块核心规范）与场景型（用户口吻）均可；若一题需多块支撑，**`gold_chunk_ids` 可含至多 3 个 id**（须均在同一法规文件内且索引连续或业务上可解释，并在 `rationale` 中简述理由）。
3. **负向约束**（写入 system / 指令）：不得虚构条号；块内无明确条号时不得编造「第×条」。
4. **每条 query 可选带 `reference`**：1～3 句中文短答，仅依据所列 gold 块可答内容，供 RAGAS Context recall 等使用。

**展开规则**：`queries.jsonl` 中**仅写入** `query` 非空的行；**0 题**的 chunk 只出现在 **`chunk_records.jsonl`**（见第 4 节），不写入 `queries.jsonl`。

## 3. 运行时配置与调用栈

### 3.1 环境变量（与实现对齐）

| 变量 | 用途 |
| --- | --- |
| **`MODEL_API_KEY`** | OpenAI 兼容 API 密钥（与线上 QA 一致） |
| **`MODEL_BASE_URL`** | 兼容端点 Base URL |
| **`GENERATE_MODEL_NAME`** | **金标/数据合成专用模型名**；**未设置**时与 **`MODEL_NAME`** 相同（代码侧使用 `Settings.generate_model_name_resolved`，见 [settings.py](../../src/conf/settings.py)） |

温度、top_p 等建议写入 **`manifest.json`**（第 4.3 节），默认 **temperature ≤ 0.3**（生成题面仍可有适度多样性）。

### 3.2 LangChain 与原生 SDK 的取舍

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| **官方 `openai` Python SDK**（与 [qa/streaming](../../src/qa/streaming.py) 一致） | 依赖已存在、行为可预期、易审计 | 需自行拼 `response_format` / JSON 解析 |
| **LangChain** `ChatOpenAI` + `with_structured_output` / LCEL | 结构化输出、链式多步扩展方便 | 新增依赖与版本锁定成本 |

**建议**：

- **默认推荐 `openai` SDK + Pydantic / JSON Schema 校验**：与本仓库 LLM 调用方式一致，适合单步「chunk → 结构化标注」。
- **若**后续需要多步（摘要 → 出题 → 交叉审查）或团队已统一 LangChain：**可**采用 LangChain，但须在 **`manifest.json`** 中声明 **`framework`: `langchain`** 及**精确版本号**，保证复现。

### 3.3 实现脚本 `e01_d05_llm_gold_generate.py`：参数说明与断点续跑

官方入口：[scripts/e01_d05_llm_gold_generate.py](../../scripts/e01_d05_llm_gold_generate.py)。**完整参数说明**以命令行为准：

```text
uv run python scripts/e01_d05_llm_gold_generate.py --help
```

脚本 `epilog` 与各参数的 `help` 为中文详解（与 `--help` 一致）。以下为常用场景与要点。

| 参数 | 作用 |
| --- | --- |
| `--d05-dir` | d05 切块导出目录（默认 `data/chunk_md/d05_heading_presplit_document_segmentation`）。 |
| `--out-root` | 金标根目录（默认 `data/eval/gold/rag_law_d05`）。 |
| `--dataset-version` | 数据集子目录名；与 **`--resume`** 联用时指向**已有**输出目录以续跑。未传则自动生成 UTC 时间戳目录名。 |
| **`--resume`** | **断点续跑**：读取已有 `chunk_records.jsonl`，按 **`chunk_id` 取最后一行**；若最新记录**无** `provenance.error`，则**跳过**该块（不调 API）。曾为 LLM/API 失败的块会**重新调用**并在文件**末尾追加**新记录（保留历史行便于审计）。同时加载 `queries.jsonl` 做 **query 文本去重**。 |
| `--name-contains` | 只处理文件名包含子串的 `.md`（如筛选「婚姻法」）。 |
| `--limit-files` / `--limit-chunks` | 调试：`--limit-chunks` 只统计**未被 resume 跳过**的块。 |
| `--temperature` / `--max-tokens` | 传给 chat.completions。 |
| `--dry-run` | 只遍历与进度演示，不调 API、不写文件；可与 `--resume` 同用以查看将跳过多少块。 |
| `--no-raw` | 不写 `raw/` 下每块原始响应。 |

**续跑与目录规则**：

- **首次**写入：`<out-root>/<dataset-version>/` 须**不存在或为空**（或目录不存在）。  
- **续跑**：同一 `--dataset-version` + **`--resume`**，且该目录下已存在 **`chunk_records.jsonl`**，则向 `queries.jsonl` / `chunk_records.jsonl` **追加**写入。  
- 若指定了 **`--resume`** 但该文件不存在，脚本会 **WARN** 并按全新运行（与首次一致）。

**`manifest.json`**：每次运行结束会重写；`counts.dataset` 为按磁盘聚合的**全量**统计（含 `chunk_ids_unique`、`queries_in_jsonl` 等）；`counts.last_run` 为**本轮**增量（含 `skipped_resume`）。

## 4. 落盘：目录、文件与企业级约定

### 4.1 根路径与版本

采用**数据集根目录 + 语义化子版本**，便于存档与评测对齐：

```text
data/eval/gold/rag_law_d05/
  └── <dataset_version>/          # 例：1.0.0 或 2026-04-09__d05__v1
        ├── README.md             # 数据卡（见下）
        ├── manifest.json         # 运行指纹与统计
        ├── schema.json           # JSON Schema URI 或内嵌 draft 版本说明
        ├── queries.jsonl         # canonical：一行一条「可评测 query」
        ├── chunk_records.jsonl   # 每个 chunk 一行：含 0 题与跳过原因
        └── raw/                  # 可选：原始 API 响应 JSON（按 chunk_id 命名，便于审计）
```

- **`<dataset_version>`**：建议 `MAJOR.MINOR.PATCH` 或 `ISO日期__短说明`；**任一**切块重导、prompt 变更、模型变更均应** bump** 新版本目录；**禁止在未使用 `--resume` 时覆盖**已有非空目录。同一版本目录的**合法追加**见第 3.3 节 **`--resume`**。
- **`README.md`（数据卡）** 至少包含：来源路径 `d05`、文件数、chunk 总数、query 总数、模型名、创建时间、git commit、已知局限。

### 4.2 `queries.jsonl`（评测主表）

每行一个 JSON 对象，**字段稳定、可版本演进**；建议在 **`manifest.json`** 中声明 **`queries_schema_version`**（如 `1.0.0`）。

**必填字段示例**：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `schema_version` | string | 本行 schema 版本，与 manifest 对齐 |
| `query_id` | string | 全局唯一，建议：`sha256(source_file + chunk_index + query_text)[:16]` 或 `{safe_stem}__c{idx}__q{n}` |
| `query` | string | 用户问题正文 |
| `gold_chunk_ids` | string[] | ES 兼容 id 列表 |
| `gold_primary_chunk_id` | string | 可选；缺省时评测脚本按 v1.1.11 取 `gold_chunk_ids[0]` |
| `reference` | string | 可选 |
| `provenance` | object | `source_file`, `chunk_index`, `annotator_model`, `prompt_version`, `created_at`（ISO 8601 UTC） |

### 4.3 `chunk_records.jsonl`（全量 chunk 审计表）

每个 chunk **至少一行**；**同一 `chunk_id` 可有多行**（如对失败块重试后追加成功记录）。解析与续跑时以 **同一 `chunk_id` 的最后一行**为准。无论是否生成问题，每次处理都会写入一行：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `chunk_id` | string | `{source_file}:{chunk_index}` |
| `source_file` | string | 同上 |
| `chunk_index` | int | 同上 |
| `n_queries` | int | 0～5 |
| `skip_reason` | string \| null | 当 `n_queries=0` 时**必填**；否则 `null` |
| `text_sha256` | string | 可选；trim 后 UTF-8 字节 SHA256，用于切块变更检测 |
| `provenance` | object | 与 queries 同构的标注元数据 |

### 4.4 `manifest.json`（运行指纹）

建议字段（企业审计常用）：

| 字段 | 说明 |
| --- | --- |
| `dataset_name` | 固定值如 `rag_law_d05_gold` |
| `dataset_version` | 与目录名一致 |
| `queries_schema_version` | 如 `1.0.0` |
| `created_at` | ISO 8601 UTC |
| `git_commit` | `git rev-parse HEAD` |
| `d05_input` | `{ "path": "data/chunk_md/d05_heading_presplit_document_segmentation", "files_sha256_manifest": "…" }`（或对目录列表的聚合哈希） |
| `llm` | `{ "model": "<GENERATE_MODEL_NAME 或 resolved>", "base_url_host": "<脱敏，仅 host>", "temperature": 0.2 }` |
| `framework` | `openai_sdk` 或 `langchain` + 版本 |
| `prompt_version` | 字符串，与代码内常量一致 |
| `counts` | 实现中分为 **`dataset`**（按 `chunk_records`/`queries` 聚合）与 **`last_run`**（本轮增量，含 `skipped_resume`）；详见脚本写出样例 |

**说明**：**不得**在 manifest 中写入 **API Key**；`MODEL_API_KEY` 仅运行时环境注入。

### 4.5 `schema.json`

- 存放 **`queries.jsonl` 单行对象**的 JSON Schema（Draft 2020-12 或 Draft-07），或 **仅含 `$id` + 指向仓库内 URL 路径**；评测 CI 可用 `jsonschema` 校验。

## 5. 结构化模型输出（建议）

为便于校验，单次 API 响应建议为**一个 JSON 对象**（非 JSONL），例如：

```json
{
  "n_queries": 2,
  "skip_reason": null,
  "items": [
    {
      "query": "……",
      "gold_chunk_ids": ["md3__宪法.md:3"],
      "gold_primary_chunk_id": "md3__宪法.md:3",
      "reference": "……"
    }
  ]
}
```

当 `n_queries=0` 时：`items` 为空数组，`skip_reason` 必填。

**校验**：`len(items) === n_queries`；`0 ≤ n_queries ≤ 5`；每条 `gold_chunk_ids` 非空且 id 合法。

## 6. 校验、去重与质控

| 环节 | 规则 |
| --- | --- |
| 结构 | Pydantic / JSON Schema 通过 |
| id | `gold_chunk_ids` 内每一 id 可解析且落在该 `source_file` 的 chunk 范围内 |
| 去重 | 跨 chunk 的 `query` 归一化后相似度超阈值的，保留一条并在 manifest `dedup_removed` 计数 |
| 人审 | 对 `n_queries≥1` 的随机 ≥5% 与 **全部** `skip_reason` 为模糊枚举的样本做人工 spot-check |

## 7. 执行顺序（建议）

1. 冻结 **d05 目录**与即将入库的 **`CHUNK_VERSION` / 索引 scope**（[v1.1.8](v1.1.8-scheme-d-eval.md)）。  
2. 配置 `.env`：`MODEL_API_KEY`、`MODEL_BASE_URL`、**`GENERATE_MODEL_NAME`**（可选）。  
3. 运行实现脚本：[`scripts/e01_d05_llm_gold_generate.py`](../../scripts/e01_d05_llm_gold_generate.py)（`uv sync --extra llm`；**`uv run python scripts/e01_d05_llm_gold_generate.py --help`** 查看全部参数说明）。流水线：枚举 chunk → 调 LLM → 写 **`chunk_records.jsonl`** 与 **`queries.jsonl`** → 写 **`manifest.json`**、**`schema.json`** 与 **`README.md`**。  
4. 人审与去重后，将该 **`dataset_version`** 目录登记为 v1.1.11 评测的**正式金标**引用路径。

### 7.1 命令示例

```bash
# 查看全部参数（含中文 epilog 详解）
uv run python scripts/e01_d05_llm_gold_generate.py --help

# 只预览：某子串文件 + 进度（不调 API）
uv run python scripts/e01_d05_llm_gold_generate.py --name-contains 婚姻法 --dry-run

# 试跑：婚姻法、最多 20 个「未 resume 跳过」的 chunk
uv run python scripts/e01_d05_llm_gold_generate.py \
  --name-contains 婚姻法 \
  --dataset-version smoke_hunyin_v1 \
  --limit-chunks 20

# 同一数据集目录续跑（换其它法规筛选或去掉 --name-contains 以继续生成未完成块）
uv run python scripts/e01_d05_llm_gold_generate.py \
  --dataset-version smoke_hunyin_v1 \
  --resume

# 全量：d05 目录下全部 .md（不按文件名筛选）
uv run python scripts/e01_d05_llm_gold_generate.py --dataset-version all_md_v1
```

## 8. 与「全量 chunk」的关系说明

- **全量**：指对 d05 目录下**每一个 chunk** 均产生 **`chunk_records.jsonl` 一行**（含 **0 题**记录）。  
- **queries.jsonl** 仅包含 **至少 1 题**的样本行数，总条数 **≤ 5 × chunk 总数**，通常远小于该上界。
